## 175. Combine Two Tables

🔗 https://leetcode.com/problems/combine-two-tables/

### 解法

```sql
SELECT
    p.firstName,
    p.lastName,
    a.city,
    a.state
FROM Person p
LEFT JOIN Address a
    ON p.personId = a.personId;
```

### 重點筆記

- 用 `LEFT JOIN` 而非 `INNER JOIN`：題目要求**所有人都要出現**，即使沒有對應地址也要列出（city 和 state 會顯示 NULL）。
- `INNER JOIN` 只會返回兩張表都有對應資料的列，會漏掉沒有地址的人。

> 💡 **口訣**：想要「左邊全部留下來，右邊沒有就 NULL」→ 用 `LEFT JOIN`

## 181. Employees Earning More Than Their Managers

🔗 https://leetcode.com/problems/employees-earning-more-than-their-managers/

### 解法

```sql
SELECT e.name AS Employee
FROM Employee AS e
JOIN Employee AS m
    ON e.managerId = m.id
WHERE e.salary > m.salary;
```

### 重點筆記

- 同一張表格 JOIN 自己，稱為 **Self JOIN**。
- 用不同的 alias（`e` = 員工、`m` = 主管）來區分兩個角色。
- `e.managerId = m.id` 是 JOIN 條件，把每個員工和他的主管連在一起。
- 最後再用 `WHERE` 篩選薪水比主管高的員工。

> 💡 **Self JOIN 使用時機**：表格內有「父子關係」或「比較關係」時（如員工與主管、類別與父類別）。

## 182. Duplicate Emails

🔗 https://leetcode.com/problems/duplicate-emails/

### 解法

```sql
SELECT email
FROM Person
GROUP BY email
HAVING COUNT(id) > 1;
```

### 重點筆記

- `GROUP BY email`：把相同 email 的列分成一組。
- `HAVING COUNT(id) > 1`：篩選出現超過 1 次的 email（即重複的）。
- **`WHERE` vs `HAVING` 的差別**：
  - `WHERE`：在分組**之前**篩選原始資料
  - `HAVING`：在分組**之後**篩選聚合結果（搭配 COUNT、SUM 等）

> 💡 **規則**：用了 `GROUP BY` 之後要再過濾 → 一定用 `HAVING`，不能用 `WHERE`。

## 183. Customers Who Never Order

🔗 https://leetcode.com/problems/customers-who-never-order/

### 解法

```sql
SELECT c.name AS Customers
FROM Customers AS c
LEFT JOIN Orders AS o ON c.id = o.customerId
WHERE o.customerId IS NULL;
```

### 重點筆記

- 先用 `LEFT JOIN` 把所有顧客保留，沒有訂單的人 `o.customerId` 會是 NULL。
- 再用 `WHERE o.customerId IS NULL` 篩選出從未下單的人。
- 注意：判斷 NULL 要用 `IS NULL`，不能用 `= NULL`（SQL 中 NULL 不等於任何值）。

> 💡 **找「不存在某關聯」的通用套路**：`LEFT JOIN` + `WHERE 右表關鍵欄位 IS NULL`

## 196. Delete Duplicate Emails

🔗 https://leetcode.com/problems/delete-duplicate-emails/

### 解法

```sql
DELETE p1
FROM Person p1
JOIN Person p2
  ON p1.email = p2.email
  AND p1.id > p2.id;
```

### 重點筆記

- 同樣是 Self JOIN，但這次是為了**刪除**重複資料。
- `p1.id > p2.id`：保留 id 較小的那筆，刪除 id 較大的重複 email。
- `DELETE p1`：只刪除 p1 的資料，p2 不受影響。
- MySQL 中 `DELETE` 可以指定刪除哪個 alias 的資料，這是 MySQL 特有語法。

> 💡 **保留最小 id 的邏輯**：若 p1.id > p2.id 且 email 相同，代表 p1 是「多餘的複本」。

## 197. Rising Temperature

🔗 https://leetcode.com/problems/rising-temperature/

### 解法

```sql
SELECT w1.id
FROM Weather w1
JOIN Weather w2
    ON w1.recordDate = w2.recordDate + INTERVAL 1 DAY
WHERE w1.temperature > w2.temperature;
```

### 錯誤紀錄與修正

❌ **第一次的錯誤答案：**
```sql
-- 錯誤：比較方向反了
WHERE w2.temperature > w1.temperature
```

✅ **正確邏輯：**
- `w1` 是「今天」，`w2` 是「昨天」（`w2.recordDate + 1天 = w1.recordDate`）
- 要找今天比昨天**熱**，所以是 `w1.temperature > w2.temperature`

### 重點筆記

- `INTERVAL 1 DAY` 是 MySQL 的日期加減語法。
- 可以用表格幫助理清方向：

| alias | 角色 | recordDate |
|-------|------|------------|
| w1 | 今天 | 較新的日期 |
| w2 | 昨天 | w1 的前一天 |

> 💡 **日期比較常用語法**：`DATEDIFF(w1.recordDate, w2.recordDate) = 1`（更通用，避免時區問題）

## 511. Game Play Analysis I

🔗 https://leetcode.com/problems/game-play-analysis-i/

### 解法

```sql
SELECT player_id, MIN(event_date) AS first_login
FROM Activity
GROUP BY player_id;
```

### 重點筆記

- `GROUP BY player_id`：每個玩家分成一組。
- `MIN(event_date)`：取該玩家最早的登入日期。
- 聚合函數（`MIN`、`MAX`、`SUM`、`COUNT`、`AVG`）必須搭配 `GROUP BY` 使用（除非對整張表聚合）。

> 💡 **找每個群組中的最小/最大值**：`GROUP BY` + `MIN()` / `MAX()`

## 今日學習重點總結

### 核心概念整理

| 概念 | 使用時機 | 範例題 |
|------|----------|--------|
| `LEFT JOIN` | 保留左表所有資料，右表沒有就 NULL | 175, 183 |
| `Self JOIN` | 同一張表有父子/比較關係 | 181, 196, 197 |
| `GROUP BY` + `HAVING` | 分組後篩選聚合條件 | 182, 511 |
| `IS NULL` | 判斷 NULL 值（不能用 `=`） | 183 |
| `INTERVAL n DAY` | MySQL 日期加減 | 197 |
| `MIN()` / `MAX()` | 取群組中的最小/最大值 | 511 |

### JOIN 類型比較

```
INNER JOIN  → 只保留兩邊都有對應的資料
LEFT JOIN   → 左邊全留，右邊沒有就 NULL
RIGHT JOIN  → 右邊全留，左邊沒有就 NULL
Self JOIN   → 同一張表 JOIN 自己（需要不同 alias）
```

### WHERE vs HAVING

```
WHERE  → GROUP BY 之前過濾（針對原始欄位）
HAVING → GROUP BY 之後過濾（針對聚合結果如 COUNT、SUM）
```
